## 1. Approach Overview

One new Admin route on the existing admin actors surface: `POST /api/v1/admin/actors/import` receiving the workbook as a **base64 JSON body** (no multipart/multer — see DR-1) with `mode: 'preview' | 'commit'`. A new `ActorImportService` parses the workbook with **exceljs**, validates each row against the same canonical rules as single create (reusing `common/normalize.ts` constants and the create-DTO bounds), classifies rows (create / skip / fail / warning), and on commit inserts in **chunked transactions** with `IMPORT` audit entries via the existing `ActorAuditService`. The template is a **committed static asset** under `frontend/public/templates/`, generated by a backend dev script from the same canonical constants (anti-drift, NFR-8). The UI is a new import view reached from the actors console toolbar.

```
Field staff ──(offline)── actor-import-template.xlsx ◄─ generated by backend/scripts/generate-import-template.ts
                                                          (exceljs, canonical constants, fixed dates → byte-stable)
Admin ─→ /admin/actors/import ('use client', RequireRole)
           1. download template (static asset link)
           2. pick .xlsx → client base64 → POST {mode:'preview'} ─→ AdminActorsController.import()
           3. render preview (create/skip/fail/warn per row)         └─ ActorImportService
           4. confirm (+ acknowledgement if file publishes)              parse (exceljs) → validate rows
              → POST {mode:'commit', acknowledged}                       → dedupe (in-file + DB)
           5. render result, refetch table                               → commit: chunked $transaction
                                                                            [createMany actors + crops + IMPORT audit]
```

## 2. Data Model Changes

- **`ActorAuditAction` enum + `IMPORT`** — additive migration `add_import_audit_action` (MySQL `ALTER TABLE ... MODIFY COLUMN action ENUM(...)`; Prisma generates it from the enum change). No table/column changes; rollback = remove the value (no rows carry it until the feature ships).
- No other schema impact. Uploads are processed in-memory per request; nothing persisted to S3/disk.

## 3. API Surface & Contracts

One route on the existing `@Controller('admin/actors')` (class-level `JwtAuthGuard` + `RolesGuard` + `@Roles('Admin')` — FR-10). Declared alongside the other collection-level routes (before `:id` routes).

| Method | Path | Body | Success | Errors |
|---|---|---|---|---|
| POST | `/admin/actors/import` | `ActorImportRequestDto` | 200 `ImportReport` | 400 (format/caps/base64), 401/403, 413 (gateway) |

**`ActorImportRequestDto`** (`class-validator`):
- `fileName: string` (`@Matches(/\.xlsx$/i)`)
- `fileBase64: string` (`@IsBase64`, decoded size ≤ **4 MB** — checked in service with a clean 400)
- `mode: 'preview' | 'commit'` (`@IsIn`)
- `acknowledged?: boolean` — file-level consent acknowledgement (FR-6), required by the service when any row sets `GRANTED` and `mode === 'commit'`.

**`ImportReport`** (both modes; `mode` echoed so the client renders "preview" vs "result"):
```ts
{
  mode: 'preview' | 'commit';
  templateVersionDetected?: string;     // from the Instructions sheet if present (best effort)
  totals: { rows: number; toCreate: number; created: number; skipped: number; failed: number; warnings: number };
  rows: Array<{
    rowNumber: number;                  // Excel data-row number (header = row 1)
    traderId: string | null;
    traderName: string | null;
    outcome: 'create' | 'created' | 'skipped-exists' | 'skipped-duplicate-in-file' | 'failed';
    actorId?: string;                   // commit + created only
    errors?: { field: string; message: string }[];   // failed only — field NAMES + messages, no PII values (FR-11)
    warnings?: string[];                // e.g. 'GPS out of range — imported with GPS cleared'
  }>;
}
```
In preview, `toCreate` counts prospective creates and `created = 0`; in commit, `created` reflects reality (FR-7). Whole-file failures (not .xlsx, corrupt, over caps, > **1,000 data rows**) → 400 with a human-readable message and `details` where field-shaped (reusing the W-1 envelope).

## 4. Backend Design

**Directory (extend `backend/src/actors/` — the import is an admin-actors capability; the legacy `src/import/` module stays untouched):**
```
actors/actor-import.service.ts        // NEW — parse, validate, classify, commit
actors/dto/actor-import-request.dto.ts// NEW
actors/actor-import.types.ts          // NEW — ImportReport, RowResult, template column map
actors/actor-import.service.spec.ts   // NEW
admin-actors.controller.ts            // +1 route
actors.module.ts                      // +ActorImportService provider
test/admin-actor-import.e2e.spec.ts   // NEW
scripts/generate-import-template.ts   // NEW (backend/scripts/) — dev-time template generator
common/template-columns.ts            // NEW — single source of truth: column defs (header, field, required, allowed values) built FROM normalize.ts constants + crop catalog
```

- **Template column map (`common/template-columns.ts`, NFR-8):** an ordered array of `{ header, field, required, allowedValues?, format? }` derived from `CANONICAL_REGIONS`, `TRADER_TYPES`, sex values, the 3-crop catalog, and `ConsentStatus`. Consumed by BOTH the generator script and the parser — headers and allowed values cannot drift. Crops are **three YES/NO columns** (`Crop: Sorghum`, `Crop: Common bean`, `Crop: Groundnut`) — dropdown-friendly for field staff, no spelling risk (DR-3). Template version = a constant (`v1`) stamped on the Instructions sheet and bumped on any column change.
- **`ActorImportService`:**
  - `run(dto, acting): Promise<ImportReport>` — decode base64 (size guard) → `exceljs` load (failures → 400 "not a valid .xlsx") → locate the Data sheet → iterate rows:
    1. **Map & normalize** via the column map + `normalize.ts` (region/type/sex normalizers, `parseCapacityTons`, GPS guards) — same quarantine philosophy as the dormant service.
    2. **Validate** with the same bounds as `AdminActorCreateDto` (required, email format, GPS ranges, consent enum). Invalid GPS → **import with GPS cleared + row warning** (detailed-design rule, OQ-4 resolved). Other rule violations → row `failed` with field errors.
    3. **Dedupe:** in-file (first `traderId` occurrence wins; later ones `skipped-duplicate-in-file`) then DB (`findMany` on the file's traderIds in one query → `skipped-exists`).
    4. **Consent gate (FR-6):** rows with `GRANTED` require `dto.acknowledged === true` on commit; otherwise those rows fail with `consentStatus: acknowledgement required`. Empty consent → `UNKNOWN`.
    5. **Preview:** stop here — no writes (FR-3).
    6. **Commit:** re-runs all of the above (stateless — no preview token), then inserts in **chunks of 100** rows: each chunk is one `$transaction` doing `actor.create` per row (crops need per-actor link rows) + one `ActorAuditService.logImport` batch (`createMany` of `IMPORT` snapshot entries, acknowledged persisted where relevant). A chunk failure rolls that chunk back, marks its rows `failed` (chunk-level error), and later chunks still run (FR-5 fault isolation).
  - Acting email resolved once, pre-transaction, via the existing `ActingAdminResolver`.
- **`ActorAuditService.logImport(tx, actors, acting, acknowledged?)`** — thin addition mirroring `logBulkConsent` batching: snapshot envelope, action `IMPORT`.
- **Controller:** `@Post('import')` + `@HttpCode(200)` (report semantics, matching bulk-delete's POST-with-200 precedent); DTO body — **no multer/multipart** (DR-1).
- **Body limit:** Express default JSON limit is ~100 kB — `main.ts`/`lambda.ts` gain `express.json({ limit: '8mb' })` via the shared bootstrap so the base64 payload fits (documented; applies to both entrypoints identically).
- **exceljs** added as a production dependency (parse) + used by the dev script (generate). Bundle/cold-start impact checked at deploy; acceptable for a single admin route (DR-2).

## 5. Frontend Design

**Directory:**
```
frontend/app/(admin)/admin/actors/import/page.tsx   // NEW 'use client' import flow
frontend/components/admin/ImportPreviewTable.tsx    // NEW — per-row outcomes (shared by preview & result)
frontend/lib/api/actors-admin.ts                    // +importActors(file → base64, mode, acknowledged?)
frontend/public/templates/actor-import-template.xlsx// NEW committed asset (generated)
frontend/app/(admin)/admin/actors/page.tsx          // toolbar: +Import link (next to New actor)
```
- **Flow (FR-9):** stepper-like single page — (1) intro + **Download template** link (static asset; contains no data, safe as public asset — OQ-5 resolved) with brief instructions; (2) file picker (`accept=".xlsx"`, client-side size guard, drag & drop optional); (3) on select → base64 → `importActors(..., 'preview')` → **preview**: totals summary chips + `ImportPreviewTable` (row #, traderId, name, outcome badge, errors/warnings expanded inline), grouped invalid-first; (4) **Confirm import** — disabled while in flight; if any previewed row publishes (`GRANTED`), the existing `AcknowledgeDialog` gates the commit and sends `acknowledged: true`; (5) result view (same table, `created` outcomes + counts, live-region announced) with "Back to actors" (table refetches on return).
- Outcome badges use the established token palette (created=positive, skipped=neutral, failed=danger, warning=caution); tables responsive (cards on mobile per console pattern); all states keyboard-accessible; errors listed per row with field names only (FR-11).
- `importActors` in `lib/api/actors-admin.ts` follows the existing typed-client conventions (`apiFetch`, Bearer, `ApiError` with field details).

## 6. Security & RBAC

- Route rides the class-level Admin guard stack (FR-10); e2e asserts 401/403/200.
- **PII handling (FR-11):** file processed in-memory only; service never logs row contents (only counts + acting sub, matching the bulk ops' structured logs); `ImportReport.errors` carry field names/messages — values never echoed. Template asset contains headers/instructions only.
- **Consent (FR-6):** server-enforced file-level acknowledgement for `GRANTED` rows on commit — same defense-in-depth as bulk unlock; recorded on the audit entries.
- Parser hardening (NFR-1): exceljs load wrapped; any parse/shape error → 400, never a crash; row cap re-checked before validation to bound CPU.

## 7. Infrastructure / Deployment

- **No template.yaml change** (existing proxy routes cover POST; payload ≤ ~5.5 MB base64 stays under the 10 MB gateway cap).
- **Migration:** `add_import_audit_action` rehearsed locally → `prisma migrate deploy` on dev RDS during the deploy task (`--profile IBD-DEV`).
- Standard backend/frontend deploys. Rollback: revert code; enum value removal only if unused.

## 8. Decision Records (ADR-style)

### DR-1: Base64 JSON upload, not multipart
- **Context:** HTTP API v2 + serverless-http; multipart needs multer, binary-media handling, and content-type negotiiation through the proxy; files are small (≤4 MB decoded). **Decision:** `{ fileName, fileBase64, mode, acknowledged }` JSON body + raised JSON body limit (8 MB) in the shared bootstrap. **Consequences:** trivially testable with supertest; one DTO validates everything; ~33% base64 overhead is irrelevant at this size. Rejected: multipart (more moving parts for zero user-visible gain); S3 presigned upload (new infra + IAM for a v1 that doesn't need it).

### DR-2: exceljs for both parse (runtime) and generate (dev script)
- **Context:** need robust xlsx read + template writing with dropdown data-validations and styled instructions. **Decision:** exceljs (prod dep for parsing; the generator is dev-time). **Consequences:** one library, one API; bundle grows ~1 MB (acceptable for the single admin Lambda; verified at deploy). Rejected: SheetJS CE (weaker write-side data-validation support in the OSS build); hand-rolled XML (absurd).

### DR-3: Crops as three YES/NO columns
- **Context (OQ-3):** comma-separated names invite typos and Excel dropdowns can't multi-select. **Decision:** one column per crop (`Crop: Sorghum` / `Crop: Common bean` / `Crop: Groundnut`) with YES/NO dropdowns; parser maps YES → crop link. **Consequences:** zero spelling risk, template-driven; adding a crop later = template version bump (explicit, versioned).

### DR-4: Single endpoint, `mode` flag, stateless commit
- **Context (OQ-2):** preview and commit share 95% of the pipeline. **Decision:** one route; commit re-validates from scratch (no preview token/state). **Consequences:** no server-side session/staleness problems (DB may change between preview and commit — commit's own dedupe decides); double parsing cost is negligible at this scale.

### DR-5: Invalid GPS → import with GPS cleared + warning
- **Context (OQ-4):** detailed-design's standing rule ("rows outside range import with GPS nulled + flagged, not plotted"). **Decision:** keep it — GPS problems are common in field data and shouldn't block a whole actor; the warning is surfaced per row and the actor simply doesn't plot. **Consequences:** a `warnings` channel exists in `ImportReport`; all other validation failures still fail the row.

### DR-6: Import lives in `actors/`, dormant `import/` module untouched
- **Context:** the legacy `ImportModule` is legal-gated scaffolding for the real source file with its exact dirty headers. **Decision:** new service beside the other admin-actors services, reusing `normalize.ts` directly; `src/import/` remains as designed for its own (still-deferred) purpose. **Consequences:** no legal-gate entanglement; clear ownership (admin actors surface); slight duplication of the "map row" idea, justified by different input shapes.

### DR-7: Template generated into `frontend/public/`, byte-stable
- **Context (OQ-5, NFR-8):** template must ship with the static export and never drift from validation. **Decision:** `backend/scripts/generate-import-template.ts` (imports `common/template-columns.ts`) writes `frontend/public/templates/actor-import-template.xlsx` with **fixed workbook created/modified dates** so regeneration without schema changes is byte-identical (CI-checkable). **Consequences:** template downloadable without auth (contains no data — accepted); one command refreshes it when constants change.

## 9. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Lambda time/memory on max file | caps (1,000 rows / 4 MB decoded) validated first; chunked commits (100); single DB query for dedupe; e2e-sized fixture test |
| Base64 body vs Express limit | shared-bootstrap `express.json({ limit: '8mb' })` — in BOTH entrypoints via the same factory pattern as the pipe (W-1 lesson) |
| exceljs bundle/cold start | measured at deploy; only route that imports it (lazy `import()` if needed) |
| PII in reports/logs | field-name-only errors; no row-content logging; e2e asserts no phone/email in report bodies |
| Template drift | single column-map module + byte-stable regeneration + version stamp; test asserts generator output matches committed asset |
| DB changes between preview and commit | stateless commit re-validates + re-dedupes; report reflects commit-time truth |
| Enum migration on live dev | additive, rehearsed, `migrate deploy` (same runbook as `ActorAuditLog`) |

## 10. Test Plan Outline

- **Unit (`actor-import.service.spec.ts`):** header/column mapping; normalization + validation per field (bad region, bad email, GPS out of range → cleared+warning); in-file dedupe (first wins); DB dedupe (mock); consent gate (GRANTED w/o ack fails, with ack passes, empty → UNKNOWN); chunking (fault isolation — failing chunk doesn't kill later chunks); report totals consistency; base64/size/row-cap guards.
- **Unit (template):** generator output byte-equals the committed asset; Instructions sheet lists exactly the allowed values from the column map; version stamp present.
- **E2E (`admin-actor-import.e2e.spec.ts`):** 401/403/200 matrix; preview writes nothing (actor count unchanged); commit lifecycle with a mixed fixture (created/skipped/failed/warning rows, audit entries visible via history route); re-upload → all skipped, 0 created (idempotence); GRANTED-without-ack → those rows fail; not-xlsx/over-cap → 400; **PII boundary + public reads regression green**; no PII values in report bodies.
- **Frontend (RTL):** flow states (pick → preview → confirm → result); ack dialog gating when preview contains GRANTED rows; outcome badges/errors render; template link present; a11y basics (labeled input, live region); client refuses non-.xlsx.
- **Gates:** backend + frontend `npm test` green; `npm run build` both; migration applies locally.
